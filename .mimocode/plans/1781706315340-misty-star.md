# Fix: Efficience Cumulée & Comparaison Inline vs Endline — Always "No Data"

## Root Causes

### 1. Efficience Cumulée ·203 — `minutes_produites` table never synced

The `efficienceTrend()` controller (`ProductionController.php:383`) queries `minutes_produites` and `minutes_presence` to compute cumulative efficiency. `SyncService::syncProduction()` (`SyncService.php:312-331`) **omits `minutes_produites`** from its sync list. The table is always empty → `eff` = 0 → flat chart = "no data".

### 2. Comparaison Inline vs Endline — Controller returns raw unpivoted rows

The frontend (`production.tsx:1537-1548`) does its own pivot: it expects raw rows where `opera === 'inline'` or `opera === 'endline'`, grouped by `shortname`. The controller (`ProductionController.php:581-586`) returns raw DB rows — but the Novacity API returns `OPERA` as operation codes (OP010, OP020), not "inline"/"endline". The frontend pivot never matches → empty chart.

### Step 2: Fix `inlineEndline()` controller — `app/Http/Controllers/Api/ProductionController.php`

Rewrite `inlineEndline()` (line 581-586) to:
1. Fetch all rows for today (with `applyFilters` support)
2. Group by `shortname` (chain name)
3. Pivot `opera` values → classify as `inline` / `endline`
4. Return `[{ chaine: "CH1", inline: N, endline: M }, ...]`

```php
public function inlineEndline(Request $request): JsonResponse
{
    $rows = DB::table('inline_vs_endline_comparison')
        ->whereDate('log_date', Carbon::today())
        ->get();

    $grouped = [];
    foreach ($rows as $row) {
        $chaine = $row->shortname;
        if (!isset($grouped[$chaine])) {
            $grouped[$chaine] = ['chaine' => $chaine, 'inline' => 0, 'endline' => 0];
        }
        $opera = strtolower(trim($row->opera));
        if ($opera === 'inline') {
            $grouped[$chaine]['inline'] += $row->count;
        } elseif ($opera === 'endline') {
            $grouped[$chaine]['endline'] += $row->count;
        }
    }

    return response()->json(['data' => array_values($grouped)]);
}
```

**Important**: If the actual `opera` values in the DB are operation codes (OP010...) rather than "inline"/"endline", we'll need to:
- Either add a classification map in the controller
- Or transform during sync

We'll verify the actual data during implementation.
