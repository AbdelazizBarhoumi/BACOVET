# Plan: Add Historical KPI Snapshot Storage for All Pages

## Context
Every dashboard page (Quality, Production, Logistics, Development) computes KPI values **live** from synced raw tables on each API call. No daily snapshots are persisted. If data changes or raw records are overwritten, historical KPI values are lost. The only exception is `manual_kpi_history` which stores monthly snapshots of manually-entered Development KPIs — but that's narrow and manual-entry-only.

## Current State

| Page | KPIs | History? |
|---|---|---|
| Quality | 16 cards (BR CGL, RFT, BR Bundling, BR GTD, BR Print, etc.) | **None** |
| Production | 8 cards (Efficience, OWE, RFT, WIP, Lost Time, BR GTD/Bundling/Print) | **None** |
| Logistics | ~10 cards (DOT, HOT, Lead Time, Stock Rotation, Occupation, etc.) | **None** |
| Development | 4 cards (RFT Dev, Livraison, Nomenclature, Réclamations) | **Partial** (`manual_kpi_history` for manual entries only) |

**Data flow**: Sync commands (`sync:quality`, `sync:production`, `sync:logistics`, `sync:drive`, `sync:gpro`) pull raw data into local tables → Controllers compute KPI values live → Frontend polls every 60s.

## Proposed Solution

### 1. New Migration: `kpi_daily_snapshots` table
A single unified table for all pages:

```php
Schema::create('kpi_daily_snapshots', function (Blueprint $table) {
    $table->id();
    $table->string('page');        // 'quality', 'production', 'logistics', 'development'
    $table->string('kpi_key');     // e.g. 'rft_jour', 'avg_efficience', 'dot', 'dev_rft'
    $table->date('snapshot_date');
    $table->decimal('value', 10, 2)->nullable();
    $table->decimal('numerator', 15, 2)->nullable();
    $table->decimal('denominator', 15, 2)->nullable();
    $table->string('status', 20)->nullable();  // green/orange/red/grey
    $table->string('source', 100)->nullable();
    $table->json('raw_data')->nullable();      // additional context
    $table->timestamp('computed_at');
    $table->timestamps();
    $table->unique(['page', 'kpi_key', 'snapshot_date']);
    $table->index('page');
    $table->index('kpi_key');
    $table->index('snapshot_date');
});
```

### 2. New Service: `KpiSnapshotService`
- `takeSnapshots(string $page)` — computes and persists all KPIs for a given page
- Uses existing `KpiComputeService` for status logic
- Called after each sync completes
- Reuses computation logic from existing controllers (extract shared methods or call controllers directly)

### 3. Update Sync Commands
After each sync, call `KpiSnapshotService::takeSnapshots()` for the relevant pages:
- `SyncNovacityQuality` → `takeSnapshots('quality')`
- `SyncNovacityProduction` → `takeSnapshots('production')`
- `SyncNovacityLogistics` → `takeSnapshots('logistics')`
- `SyncGoogleDrive` → `takeSnapshots('quality')` + `takeSnapshots('development')` (Drive feeds both)
- `SyncNovacityFull` → `takeSnapshots('*')` (all pages)

### 4. New API Endpoint: `GET /history/{page}/{kpiKey}`
- Returns historical snapshots for a given page + KPI key
- Query params: `from`, `to`, `limit` (default 30 days)
- Used by KpiDetailModal to show historical trend

### 5. Frontend: Update KpiDetailModal
- When user clicks a KPI card, fetch from `/history/{page}/{kpiKey}`
- Show a line chart of historical values
