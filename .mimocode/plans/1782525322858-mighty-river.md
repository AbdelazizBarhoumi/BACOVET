### ❌ FAILING — 6 Violations Found

#### VIOLATION 1 — Rule #1: GaugeChart hardcodes status (DevelopmentController hardcoded source label)

**File:** `resources/js/pages/development.tsx:85`
```tsx
const color = v >= target ? 'var(--success)' : v >= target - 3 ? 'var(--warning)' : 'var(--destructive)';
```

**Problem:** The `GaugeChart` component computes color independently instead of using the `status` from the API response. This duplicates logic and can drift from the backend.

**Fix:** Pass `status` prop to `GaugeChart` from the API response and use it for color mapping. Remove the inline ternary.

---

#### VIOLATION 2 — Rule #6: Réclamations status thresholds mismatch (backend vs config vs spec)

**Files:** `DevelopmentController.php:39-41` vs `devKpiDetailConfig.ts:148-151`

| Source | Green | Orange | Red |
|--------|-------|--------|-----|
| **Spec** (finalspecs/dev.md) | < 2% | 2%–3% | > 3% |
| **Config** (devKpiDetailConfig.ts) | < 2% | 2%–3% | > 3% |
| **Backend** (target=2, target_kind='max') | <= 1% | <= 2% | > 2% |

**Concrete mismatch:**
- Value 1.5%: Config says GREEN (< 2%), Backend says ORANGE ❌
- Value 2.5%: Config says ORANGE (2%–3%), Backend says RED ❌

**Fix:** Change backend status logic for `target_kind='max'`:
```php
// BEFORE (wrong):
$status = $value <= $meta['target'] - 1 ? 'green'
    : ($value <= $meta['target'] ? 'orange' : 'red');

// AFTER (matches spec):
$status = $value < $meta['target'] ? 'green'
    : ($value <= $meta['target'] + 1 ? 'orange' : 'red');
```

This gives: green < 2%, orange 2%–3%, red > 3% — matching spec and config.

---

#### VIOLATION 3 — Rule #7: No staleness indicator on data source

**File:** `DevelopmentController.php:53-55`

The API returns `source` as a string (`'sync_drive_development'` or `'manual_kpi_values'`) and `updated_at` is only populated from the manual fallback. When using drive data, there's no way to tell if the data is fresh or stale.

**Fix:**
- Always return `synced_at` per-KPI from the source table
- Add `is_stale` boolean (true if `synced_at` is older than 7 days for monthly data)
- Return both `synced_at` and `is_stale` in each KPI object
- Show staleness indicator in the frontend source label

---

#### VIOLATION 4 — Rule #8: Config source table doesn't match actual primary source

**File:** `devKpiDetailConfig.ts:59,91,123,155`

All 4 KPIs show `mysqlTable: 'manual_kpi_values (dev_*)'` but the actual primary source is `sync_drive_development`. The config should reflect the intended primary source.

**Fix:** Change `mysqlTable` in each KPI config to `'sync_drive_development'` with a note that `manual_kpi_values` is the fallback.

---

#### VIOLATION 5 — Rule #9: synced_at is global, not per-KPI

**File:** `DevelopmentController.php:60-61`

The response returns a single `synced_at` at the top level:
```php
'synced_at' => DB::table('sync_drive_development')->orderByDesc('synced_at')->value('synced_at')
    ?? ManualKpiValue::max('updated_at'),
```

All 4 KPIs share this one timestamp. Per Rule #9, each card should carry its own `synced_at`.

**Fix:** Since all 4 KPIs come from the same `sync_drive_development` table, the `synced_at` is the same for all — but it should be returned per-KPI for consistency with other pages. Move `synced_at` into each KPI object.

---

#### VIOLATION 6 — Rule #10: No E2E verification script for development page

**Missing:** `scripts/verify_development_kpis.php`

Other pages have E2E scripts (logistics: `verify_logistics_kpis.php`, quality: `quality_e2e_check.php`).

**Fix:** Create `scripts/verify_development_kpis.php` following the pattern of `verify_logistics_kpis.php` with:
1. Table existence check (`sync_drive_development`, `manual_kpi_values`)
2. Data freshness check (synced_at age)
3. Calculation verification (recompute each KPI from raw data and compare)
4. Status threshold boundary tests (green/orange/red boundaries)
5. Rule compliance checklist
