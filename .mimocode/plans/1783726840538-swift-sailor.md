# Fix: Data Page Auth Redirect + Return-URL After Login

## Problem

The `/v1/data` page has no auth redirect when unauthenticated. Other v1 pages redirect to `/login` on 401 via their API services, but `dataMappingApi.ts` lacks this. Additionally, no v1 page has a return-URL mechanism — after login, users always land on their role's default page.

The previous attempt used `?returnTo=` query params, but `/login?returnTo=...` returns 404 because the Fortify GET `/login` route isn't registered. Fix: use `sessionStorage` instead of query params.

## Changes (2 files)

### 1. `resources/js/services/dataMappingApi.ts` — Add 401/403 handling + sessionStorage

Replace lines 29-31 (`if (!response.ok)` block) with:

```typescript
if (response.status === 401) {
    sessionStorage.setItem('v1_returnTo', window.location.pathname);
    window.location.href = '/login';
    throw new Error('Non authentifié');
}
if (response.status === 403) {
    window.location.href = '/unauthorized';
    throw new Error('Accès refusé');
}
if (!response.ok) {
    throw new Error(`API Error: ${response.status} ${response.statusText}`);
}
```

### 2. `resources/js/pages/auth/login.tsx` — Read sessionStorage after login

Replace lines 98-99 with:

```typescript
const returnTo = sessionStorage.getItem('v1_returnTo');
sessionStorage.removeItem('v1_returnTo');
const redirectTo = (returnTo && returnTo.startsWith('/v1/'))
    ? returnTo
    : (payload?.redirect ?? '/dashboard');
router.visit(redirectTo);
```

## Flow After Fix

1. Unauthenticated user visits `/v1/data`
2. SPA loads, `fetchMappings()` fires → API returns 401
3. `sessionStorage.setItem('v1_returnTo', '/v1/data')` → redirect to `/login`
4. User logs in → login page reads `sessionStorage` → navigates to `/v1/data`
