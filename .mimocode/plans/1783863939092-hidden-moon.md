# Fix: Test column stuck spinning — race condition with `startTransition`

## Root Cause
The test useEffect (line 1071) sets `testLoading[r.id] = true` via `React.startTransition` (line 1084), which defers the state update. When the effect re-runs (e.g., after `updateImmediate` changes `rows`), it reads `testLoading[r.id]` and filters out rows that are "loading" (line 1080). 

The race condition:
1. Effect run 1: `setTestLoading(r.id, true)` via `startTransition` → commits
2. `rows` changes → effect run 2: reads `testLoading[r.id] === true` → row filtered OUT → no new fetch
3. Cleanup of run 1: aborts old fetch → catch block returns without resetting `testLoading` (line 1121)
4. `testLoading[r.id]` stays `true` forever → spinner never stops

## Fix
**File:** `resources/js/routes-v1/pages/data.tsx`

Replace the `testLoading` state-based check with a ref-based tracking:

1. Add a `fetchingRef` at line ~897 (near other refs):
```ts
const fetchingRef = useRef<Set<number>>(new Set());
```

2. In the test useEffect (line 1074-1081), replace `!testLoading[r.id]` with `!fetchingRef.current.has(r.id)`:
```ts
const targets = rows.filter((r) => {
  if (!r.endpoint || !r.variable_key) return false;
  const aggSignature = ...;
  const prevSignature = prevAggSettingsRef.current[r.id];
  const needsRecompute = !testValues[r.id] || aggSignature !== prevSignature || testValues[r.id] === "Erreur" || testValues[r.id] === "Pas de sample";
  prevAggSettingsRef.current[r.id] = aggSignature;
  return needsRecompute && !fetchingRef.current.has(r.id);
});
```

3. In the `fetchInBatches` callback (line 1083-1127), add/remove from `fetchingRef`:
```ts
fetchInBatches(targets, BATCH_SIZE, async (r, signal) => {
  fetchingRef.current.add(r.id);
  React.startTransition(() => setTestLoading((m) => ({ ...m, [r.id]: true })));
  try {
    // ... existing fetch logic ...
  } catch (e) {
    // ... existing error handling ...
  } finally {
    fetchingRef.current.delete(r.id);
  }
}, controller.signal);
```

4. In the cleanup (line 1128), also clear fetchingRef for aborted rows:
```ts
return () => {
  controller.abort();
  for (const r of targets) {
    fetchingRef.current.delete(r.id);
  }
};
```

This way, the "is currently fetching" check uses an immediate ref instead of deferred state, eliminating the race condition.

## Verification
1. Select an endpoint → test column shows loading → after fetch completes, shows value
2. Quickly change endpoint again → previous fetch aborts, new one starts → eventually shows value
3. No permanent spinning state
