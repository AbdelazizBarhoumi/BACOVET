# Plan: Parallel Instant Endpoints via Queue

## Problem

The instant schedule runs synchronously — 95+ sequential HTTP requests × 30s timeout = 49 minutes per run. Each endpoint waits for the previous one to complete.

## Fix

Use `--queue` for instant endpoints too. The scheduler dispatches all instant jobs to the queue every minute. Queue workers process them in parallel. Multiple workers = multiple concurrent requests.

## Changes

### 1. `routes/console.php` — Change instant to use `--queue`

```php
Schedule::command('sync:kpi-endpoints', ['--frequency=instant', '--queue'])
    ->everyMinute()
    ->name('kpi-instant')
    ->withoutOverlapping(2);
```

### 2. No other changes needed

The `dispatchByFrequency()` method already dispatches `SyncKpiEndpointJob` to the queue. The `--queue` flag makes the command use this path. Queue workers pick up jobs and execute them concurrently.

## How it works

1. Scheduler fires every minute → dispatches ~95 jobs to the queue (fast, ~1 second)
2. Queue workers process jobs in parallel (default concurrency = number of CPUs)
3. Each job makes one HTTP request and writes to DB
4. Frontend polls DB every 1 second → picks up new data instantly

## Verification

- Check `php artisan queue:work` is running (or use `queue:listen`)
- Watch scheduler log — dispatch should take ~1s, not 49 minutes
- Dashboard should show updated data within seconds of a sync cycle
