# Fix: Dynamic Fetch — Prevent Tight Loop at Scheduler Boundary

## Root Cause

When the scheduler and frontend fetch both fire at ~T=60, there's a race:
1. Frontend fetches at T=60 → sees data from T=0 (age=60s) → delay=1s
2. T=61: fetch → still same data → retry 5s
3. T=66: fetch → scheduler ran at T=60 → data is 6s old → delay=54s
4. Total gap: 66 seconds. Counter shows 1:06.

The delay is recalculated correctly, but the **gap between fetches** (the tight loop period) eats into the 60-second window.

## Fix

Add a `lastFetchAtRef` that tracks when the last fetch happened. After each fetch, enforce `delay = max(delay, 10_000 - elapsed)` — never fetch again within 10 seconds. This prevents the tight loop while keeping dynamic scheduling.

**File:** `resources/js/pages/kpi-endpoints.tsx`

### 1. Add ref (after line 109)
```tsx
const lastFetchAtRef = useRef<number>(Date.now());
```

### 2. Update scheduleNextPoll (line 123-142)
```tsx
const scheduleNextPoll = useCallback((data: KpiEndpointRow[]) => {
    clearPollTimer();
    if (!mountedRef.current) return;

    const latest = data.reduce((max, r) =>
        r.last_synced_at && r.last_synced_at > max ? r.last_synced_at : max
    , '');

    if (!latest || latest === lastSyncedAtRef.current) {
        pollTimerRef.current = setTimeout(runPoll, 5_000);
        return;
    }

    lastSyncedAtRef.current = latest;
    const ageMs = Date.now() - new Date(latest).getTime();
    const elapsed = Date.now() - lastFetchAtRef.current;
    // Dynamic delay: when data hits 60s, but never within 10s of last fetch
    const delay = Math.max(10_000 - elapsed, 1_000, 60_000 - ageMs);
    pollTimerRef.current = setTimeout(runPoll, delay);
}, []);
```

### 3. Update runPoll (line 144-158)
```tsx
const runPoll = useCallback(async () => {
    if (!mountedRef.current) return;
    lastFetchAtRef.current = Date.now();
    // ... rest unchanged
}, [scheduleNextPoll]);
```

### 4. Update fetchRows (line 160+)
Also set `lastFetchAtRef.current = Date.now()` in `fetchRows` so manual refresh/fire also resets the timer.

## Result

| Scenario | Before Fix | After Fix |
|----------|-----------|-----------|
| Data 33s old | Fetch in 27s | Fetch in 27s (same) |
| Scheduler race at T=60 | Tight 1s loop → 66s gap | 10s minimum → max 56s gap |
| Data just synced (0s old) | Fetch in 60s | Fetch in 60s (same) |
