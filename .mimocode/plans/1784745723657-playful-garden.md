## 2. All DB Data Types

### KpiResult (from `computed_result` JSON in `kpi_data` table)

```typescript
type KpiResult = {
  scalar_value: number | null;    // The single computed number
  status: string;                  // "green" | "orange" | "red" | "grey"
  mapped_rows: Record<string, unknown>[] | null;  // Array of row objects
  filter_options: Record<string, string[]>;        // Available filter values
  computed_at: string | null;      // ISO 8601 timestamp
};
```

### KPI Computation Modes (backend)

| Mode | When | `scalar_value` | `mapped_rows` | Example KPI |
|------|------|---------------|---------------|-------------|
| **Scalar formula** | 2+ vars, aggregated (Direct type or has function) | Computed number | **null** | F-REQ-102: `Total_rejetes / Total_colis * 100` |
| **Row-by-row formula** | Complex type + no aggregation + multiple rows | Average of row values | `[{joinKey: "x", value: 12}, ...]` | Charts with per-row breakdown |
| **Single variable** | 1 variable, no formula | Aggregated value (Latest/Sum/Avg/etc) | **null** | Simple count, latest value |

### What `mapped_rows` contains (when not null)

Each row is an object like:
```json
{ "category": "A", "value": 42 }
```
- One key is always `value` (the computed number)
- Other keys are join/filter keys (category names, dates, etc.)
- The first non-`value` key is used as the x-axis label in charts

## 4. Problem Matrix — What Breaks and Why

### Problem 1: Scalar KPI + Chart widget = EMPTY chart

**Situation:** User picks a scalar-only KPI (e.g., F-REQ-102) and assigns it to a `line`, `bar`, `sparkline`, etc.

**What happens:** `mapped_rows` is `null` → `resolveKpiSeries()` returns `[]` → chart renders empty.

**Why:** The backend only computes `mapped_rows` for "Complex" variable types with multiple rows. Most KPIs are scalar formulas (single aggregated number).

**Fix options:**
- A) Hide chart widget types when KPI has no series data (like we did for sparkline toggle)
- B) Generate synthetic series from historical data (requires new backend endpoint)
- C) Show a "no data" message in the chart area

### Problem 2: Series KPI + KPI card = shows average, not the intended value

**Situation:** User picks a series KPI and assigns it to a `kpi` card.

**What happens:** `resolveKpiSeries` returns data → big number shows average of all series values.

**Why:** We compute `series.reduce(sum) / series.length` when series exists.

**Is this correct?** Depends on the KPI. For "BR GTD" (defect rate), showing the average across categories makes sense. For "total production", it doesn't.

**Fix options:**
- A) Always show `scalar_value` from DB (the backend already computes this correctly) — remove the series-average override
- B) Let the user choose: "Show scalar" vs "Show average of series"
- C) Keep current behavior (average) since backend already puts the right scalar in `scalar_value`

### Problem 3: No KPI code assigned = everything shows 0

**Situation:** User creates a widget but doesn't pick a KPI code.

**What happens:** `kpiData?.get(undefined)` → undefined → `hasData = false` → shows 0 or "—".

**Why:** No data source configured.

**Fix options:**
- A) This is correct behavior — user must pick a KPI
- B) Show a more visible "No KPI selected" state instead of 0

### Problem 4: KPI picker vs manual KPI code = two input methods

**Situation:** Inspector has both a KPI picker (Select dropdown) AND a manual "Code KPI" text input.

**What happens:** User can set `kpiCode` via either method. The picker also auto-fills label and target. The manual input doesn't.

**Why:** The picker is the primary method; the manual input is a legacy fallback.

**Fix options:**
- A) Remove the manual "Code KPI" input — only use the picker
- B) Keep both but make manual input hidden when picker is used

### Problem 5: `variable` and `aggregation` config keys are unused

**Situation:** Inspector exposes "Variable JSON" and "Agrégation" inputs, but no widget reads them.

**What happens:** Values are saved to config but never used by any widget renderer.

**Why:** These are leftover from an earlier design where widgets fetched their own data. Now all data comes through `kpiCode` → `useKpiData` → `resolveKpiValue`/`resolveKpiSeries`.

**Fix options:**
- A) Remove these inputs entirely
- B) Wire them up to actual functionality

### Problem 6: `status` color is DB-driven but not configurable

**Situation:** KPI card color is determined by `kpiResult.status` ("green"/"orange"/"red") or target comparison. User can only set `accent` color which is NOT used by KPI card.

**What happens:** The `accent` color picker exists for KPI but the widget ignores it — it uses status-based colors (#22c55e, #f59e0b, #ef4444).

**Fix options:**
- A) Remove accent color from KPI widget (it's misleading)
- B) Use accent color as override when set, fall back to status color

## 6. Status-Based Coloring Across All Widgets

### Problem
Only the KPI card changes color based on target comparison. Gauge always shows the accent color (or a percentage-based fallback). Charts use a single color for all bars/points.

### Desired Behavior
All widgets should use the same status logic as the KPI card:
- **Green** (`#22c55e`): value >= target
- **Orange** (`#f59e0b`): value >= target * 0.9 (near target)
- **Red** (`#ef4444`): value < target * 0.9

When no target is set, use the accent color (or default blue for charts).

### Status Color Helper (add to shared.tsx)

```typescript
export function statusColor(value: number, target?: number): string {
  if (!target) return "";  // no target = no status color, use accent
  if (value >= target) return "#22c55e";
  if (value >= target * 0.9) return "#f59e0b";
  return "#ef4444";
}
```

### Widget Changes

#### gauge.tsx
**Current:** `const color = c.accent ?? (pct >= 80 ? "#10b981" : pct >= 50 ? "#f59e0b" : "#ef4444")`
**New:** Use `statusColor(value, c.target)` when no accent is set. When accent IS set, still use accent (user override).

```typescript

```

#### bar-chart.tsx (uses BarKpi from v1/primitives.tsx)
**Current:** Passes single `color` prop to `BarKpi`.
**New:** Compute per-bar colors based on target and pass them via `data[].color`.

The `BarKpi` component already supports per-bar colors — each item in `data` can have a `color` property. Just map the series:

```typescript
const coloredSeries = series.map((s) => ({
  ...s,
  color: c.target ? statusColor(s.v, c.target) : undefined,

```

#### line-chart.tsx
**Current:** Single `color` for the whole line.
**New:** Use ECharts `visualMap` to color each data point based on target. When target is set, each point gets its own color. When no target, use accent color.

Approach: Use `itemStyle` per data point + `symbolColor` callback. ECharts supports per-point styling via the `data` array:

```typescript

  value: v,
  itemStyle: { color: c.target ? statusColor(v, c.target) : c.accent ?? "#3b82f6" },
})),
```

#### area-chart.tsx
Same approach as line-chart — per-point `itemStyle` coloring.

#### combo-chart.tsx
The bar series in combo chart: same approach as bar-chart — per-bar `itemStyle` coloring.

#### donut.tsx
Uses `resolveKpiValue` which already returns status. The donut ring color should match the status. Currently uses fixed `DONUT_COLORS` array. Change to use status color for the main segment.

#### simple-table.tsx
Color the value column text based on status per row.

# KPI COMPUTATION PIPELINE — FULL ANALYSIS

## Overview

The KPI system has 4 stages:
1. **Config** (`config/data-mappings.php`) — defines KPI formula, variables, types, targets
2. **External API** — fetches raw data from Novacity endpoints
3. **Storage** (`kpi_data` table) — stores raw data and computed results
4. **Computing** (`KpiResultComputer`) — computes `scalar_value` and `mapped_rows`

## Variable Types

### Direct
- Single value per variable (e.g., `{Total_rejetes: 15}`)
- API returns a flat object with one row
- `variable_key` identifies which field to extract

### Complex
- Multiple rows per variable (e.g., 100 employees, 33 chains)
- API returns an array of objects
- `filter_key` identifies the join/grouping key (e.g., `EmployeeNo`, `ProdGroup`)

## Aggregation Functions

When `has_function=true` or `fn != "Latest"`:
- **Latest** — last value in the array
- **Sum** — sum of all values
- **Average** — mean of all values
- **Min** / **Max** — extremes
- **Count** — number of rows
- **First** — first value

## Computation Paths

### Path 1: Scalar Formula (Direct + no aggregation)
**When:** All variables are Direct type, no aggregation, formula exists
**Process:**
1. For each variable, call `aggregateRaw()` to get a single value
2. Feed values into `computeFormulaScalar()` to compute the result
**Result:** `scalar_value` = computed number, `mapped_rows` = null

**Example — F-REQ-102 (BR GTD):**
```
Config: Variables=[Total_rejetes, Total_colis], Formula=var0/var1*100
Raw: Total_rejetes=15, Total_colis=183
Compute: 15/183*100 = 8.20
scalar_value=8.20, mapped_rows=null, status=red (target <=5)
```

### Path 2: Row-by-Row Formula (Complex + no aggregation + multiple rows)
**When:** At least one variable is Complex type, no aggregation, multiple rows in raw data
**Process:**
1. Find join key (prefer `filter_key`, then shared keys not in variable_keys)
2. Build lookup from first variable's raw data
3. For each row in last variable, match by join key
4. Extract each variable's value from the matched rows
5. Compute formula per row
**Result:** `scalar_value` = average of all row values, `mapped_rows` = array of row results

**Example — F-REQ-201 (Efficience par OPÉRATEUR):**
```
Config: Variables=[MinuteProduite, TempsPresence_Min], both Complex
        Formula=var0/var1*100, filter_key=EmployeeNo
Raw: 100 employees, each with MinuteProduite and TempsPresence_Min
Join: on EmployeeNo
Compute per row: 
  Employee 6328: 129.6/0 = null (÷0)
  Employee 6374: 234/6*100 = 3900
  Employee 6388: 0/423*100 = 0
  ...
scalar_value = average of all valid row values = 704.92
mapped_rows = [{EmployeeNo:"6328", value:null}, {EmployeeNo:"6374", value:3900}, ...]
status = green (target >=90, but 704.92 >= 90)
```

### Path 3: Aggregated Formula (Complex + has aggregation)
**When:** Variables are Complex but have aggregation (has_function=true or fn != "Latest")
**Process:**
1. For each variable, call `aggregateRaw()` to get a single value
2. Feed values into `computeFormulaScalar()`
**Result:** `scalar_value` = computed number, `mapped_rows` = null

**Example — F-REQ-205 (WIP par chaine):**
```
Config: Variables=[WIP_Chaine(Complex,Sum), Direct]
        Formula exists but uses Sum aggregation
Raw: WIP_Chaine has 33 rows (one per chain)
Compute: aggregateRaw(Sum) → sum of all WIP values
scalar_value = -728, mapped_rows = null
```

### Path 4: Single Variable (1 variable, no formula)
**When:** Only one variable, no formula
**Process:**
1. Call `aggregateRaw()` on the single variable
**Result:** `scalar_value` = aggregated value, `mapped_rows` = null

**Example — F-REQ-211 (SAM par chaine):**
```
Config: 1 variable (TempsStandard, Complex, filtered by ProdGroup)
Raw: 33 rows (one per chain)
Compute: aggregateRaw(Latest) → last value
scalar_value = 1316.6, mapped_rows = null
```

## Decision Tree in KpiResultComputer

```
computeKpi(kpiDef):
  1. Gather raw_data for each variable
  2. Extract filter_options
  
  IF formula exists AND >= 2 variables:
    IF any variable is Complex AND no aggregation AND has multiple rows:
      → ROW-BY-ROW PATH (Path 2)
      → mapped_rows = computeRowByRow()
      → scalar_value = average of mapped_rows values
    ELSE:
      → SCALAR FORMULA PATH (Path 1)
      → scalar_value = computeFormulaScalar(aggregated values)
  ELSE:
    → SINGLE VARIABLE PATH (Path 4)
    → scalar_value = aggregateRaw(first variable)
  
  3. Compute status from scalar_value vs target
  4. Store on leader variable's row
```

## Status Computation

```php
computeStatus(value, operator, target):
  if value is null → grey
  if no operator or target → green
  
  '<=' → value <= target ? green : (value <= target*1.1 ? orange : red)
  '>=' → value >= target ? green : (value >= target*0.9 ? orange : red)
  '<'  → value < target ? green : red
  '>'  → value > target ? green : red
  '='  → value == target ? green : red
```

**Frontend** (`safeMathEval` in `data.tsx:559-607`): Has proper operator precedence via recursive descent parser:

private function buildExpression(array $items, array $variableValues): ?string
{
    $parts = [];
    $varIndex = 0;
    foreach ($items as $item) {
        if ($item['type'] === 'variable') {
            if ($varIndex >= count($variableValues)) return null;
            $val = $variableValues[$varIndex++];
            if ($val === null) return null;
            $parts[] = (string) $val;
        } elseif ($item['type'] === 'operator') {

        } elseif ($item['type'] === 'number') {
            $parts[] = (string) $item['value'];
        } elseif ($item['type'] === 'lparen') {
            $parts[] = '(';
        } elseif ($item['type'] === 'rparen') {
            $parts[] = ')';
        }
